# 文档写作

## 目的

这份文档描述当前实现对“创建或更新企微文档”的硬约束。

## 默认输出方式

当前系统 prompt 的默认策略是：

- 用户说“总结”“整理”“概括”时，优先写入企微文档
- 用户明确说“告诉我”“简单说一下”时，才直接在聊天里给文字

所以默认不是“先在聊天里写长文，再决定要不要建文档”，而是优先落到文档。

## 新建文档的写入顺序

当前必须遵守：

1. 先调用外部 MCP 的 `create_doc`
2. 再继续调用正文写入动作
3. 只有正文真正写入成功后，才能告诉用户“已完成”

原因：

- `create_doc` 只会创建空文档壳
- 不会自动写正文

## 编辑已有文档的默认策略

当前默认优先编辑已绑定文档。  
只有用户明确要求“重新生成 / 新建一份”时，才不复用当前绑定文档。

文档续写或改写时，当前可用的本地工具是：

- `doc__read_markdown`
- `doc__append_section`
- `doc__preview_replace`
- `doc__replace_section`
- `doc__expand_section`

## 章节级写入的实际做法

### `doc__append_section`

用途：

- 在文档末尾追加一节
- 新增一个章节
- 或根据相关章节插入一个下级小节

### `doc__preview_replace`

用途：

- 先读出当前最像目标的章节
- 给替换动作提供 section payload

### `doc__replace_section`

用途：

- 用新的 Markdown 章节替换最相关 section

### `doc__expand_section`

用途：

- 在相关章节下扩写一个新小节
- 或直接新增一个顶层章节

## Markdown 约束

当前本地文档工具会做两件事：

1. 去掉正文开头与标题重复的 heading
2. 去掉连续重复的 heading

因此写正文时要注意：

- 不要把文档标题再写成正文第一行 H1
- 章节正文应该从内容本身开始，而不是再重复一次 `## 标题`

## 读文档的边界

`doc__read_markdown` 是内部编辑辅助能力，不是“给用户展示全文”的能力。

也就是说：

- agent 可以为了编辑而读取文档 Markdown
- 但不能把读取到的全文直接当聊天回复发给用户

## 写作风格的当前约束

正文应优先满足这些要求：

- 标题清晰
- 段落短
- 能快速扫描
- 不编造没有来源支撑的结论

如果涉及多份来源材料：

- 可以综合总结
- 但不要跳过用户明确要求覆盖的材料

## 当前禁止的做法

- 只创建文档壳，不写正文，就回复“已经完成”
- 把应该写进文档的正文直接发在聊天里代替写入
- 用导出 PDF 或其它无关工具冒充“文档已更新”
- 在失败或未完成时说“已写入”
