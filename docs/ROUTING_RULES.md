# 路由规则

## 目的

这份文档定义：

- 输入先按什么维度分流
- 什么情况下先澄清
- 什么情况下直接执行
- 什么情况下必须确认后再执行

## 两层分流

### 第一层：消息载体类型

优先区分：

1. 文本消息
2. 链接
3. PDF
4. 图片
5. 其他

说明：

- `链接` 主要指文本消息里包含的链接，尤其是企业微信文档链接
- `PDF` 指文件上传消息
- `图片` 当前阶段不是重点能力，但仍应先识别为图片消息，而不是误当成文本或知识库材料

### 第二层：文本语义类型

只在“文本消息”里继续细分：

1. 普通文本
2. 包含企业微信文档链接的文本
3. 包含其他链接的文本
4. 纯指令文本

## 默认主流程

- 当前默认采用：
  - agent 先输出结构化 `intent packet`
  - 规则和关键词只做辅助校验，不做主路由

- 文本消息
  - 先让 agent 理解
- 企业微信文档链接
  - 先判断是不是企业微信文档链接，再判断用户意图
- PDF
  - 先入知识库
- 图片
  - 当前阶段必要时先澄清
- 其他
  - 先澄清

## 企业微信文档链接

### 链接 + 意图不明确

如果用户给了企业微信文档链接，但意图不明确，默认先澄清，不直接编辑、总结或重写。

例如：

- “看一下这个”
- “这个怎么弄”
- “帮我处理一下”

### 澄清方式

默认采用“半结构化 + 开放补充”：

- 先给 2 到 4 个高频意图选项
- 再给一个开放补充口子
- 不要求用户只能回复编号

推荐选项：

1. 总结这份文档
2. 修改这份文档
3. 读取这份文档的主要内容

“读取主要内容”指给结构概览或简短摘要，不是再次只发链接，也不是默认贴全文。

### 链接与上下文

即使消息里有企业微信文档链接，系统也只能把它当候选对象，不能替代明确意图。

系统在澄清前，应先检查：

- 当前会话里是否已有绑定文档
- 当前链接是否就是当前绑定文档
- 最近一次用户意图是什么

如果意图仍不明确，默认先澄清。

## PDF 上传

### 用户只上传 PDF，不说话

默认流程：

1. 先确认文件已加入知识库
2. 再主动询问下一步意图
3. 说明应使用用户能懂的话，不讲技术实现细节

### 上传后只说“帮我总结一下”

默认不擅自决定输出形式，先澄清：

- 直接在聊天里总结
- 生成一份企业微信文档
- 两者都要

### 上传后明确说“总结并生成文档”

如果用户明确说：

- “帮我总结一下，并生成文档”
- “把刚上传的 PDF 总结一下，并生成文档”

则视为意图明确：

1. 不再额外澄清
2. 先确认文件已可用
3. 直接执行总结 + 生成文档

## 知识库文件查询

### 询问知识库里有几篇文章

默认返回：

- 总数量
- 按顺序列出前 10 篇文件名

如果用户明显只想要简略答案，可以只给总数，并说明如有需要可继续列出前 10 篇。

不默认附带简介；只有用户继续追问时，再补每篇简介或摘要。

### 列知识库文件

默认把固定知识库文件和用户上传文件混在一起列出。

默认不额外给上传文件打标。

只有用户继续追问“哪个是上传文档”时，再补充说明。

如果系统刚刚已经列过部分文件，而用户继续强调：

- “全列出来”
- “所有文件都列出来”
- “不是前三篇，是全部”

则默认沿用上一次知识库列表语义，直接返回完整文件列表，不再回退到泛化澄清。

### “第 N 份文件”

如果用户在上一轮已经拿到知识库候选列表后继续说：

- “第 4 份文件”
- “修改一下第 4 份文件的名字”

默认应：

1. 先从最近一次知识库候选列表里解析序号引用
2. 再按文件管理意图继续走 `kb.*`

当前阶段不应：

- 把这类序号引用改名请求回退到默认自由问答
- 把文件管理意图错误转成 `rag.*`

### “哪个是我刚上传的文档”

如果最近有明确上传记录：

- 默认只回答最近一个上传文件

如果最近没有明确上传记录：

- 列出可能候选文件给用户确认

### “我之前上传过哪些文件”

如果用户明确在问自己上传过的文件：

- 默认只返回上传文件集合
- 不混入固定知识库材料
- 如果上传文件很多，后续也支持继续追问“把所有上传文件都列出来”

### “有没有和 XX 相关的文档”

默认流程：

1. 先做文件名 / 元信息匹配
2. 如果完全没有候选，或候选太多、置信度不够，再进入 RAG 检索

RAG 找到相关文档后：

- 默认先列候选
- 再让用户决定下一步操作

如果用户问的是非常泛的问题，例如：

- “知识库里有没有讲这个方向的材料？”

默认先给一句模糊引导式判断，再列候选，不先下过于绝对的结论。

### “帮我找最适合回答这个问题的文档”

默认返回：

- 前 3 个候选
- 并说明最推荐哪个

每个候选默认带：

- 文件名
- 一句相关性说明

系统说明“最推荐的是 X”时，默认给一个简短理由。

### 候选后的下一步

如果系统已经列出候选文档：

- 默认只给一个开放式引导
- 例如：“请告诉我你想针对哪一篇文档做什么操作”

如果用户明确按序号选择，例如：

- “就看第 2 个”

则默认直接进入下一步处理，不再要求重新确认编号。

如果用户只说：

- “这个”

则默认只引用当前候选列表里的第一个候选，并向用户确认；不能按系统内部最近提到的其他候选自由猜。

## 知识库文件改名

### “你能够给这些文件改名吗”

如果用户在知识库文件语义下询问是否支持改名：

- 默认保持在知识库文件管理语义内回答
- 不应误转成企微文档标题修改
- 应明确说明当前支持范围和下一步怎么做

### “把某个文件改名为新名字”

如果用户明确给出：

- 原文件名
- 新文件名

默认先进入确认步骤，不直接改名。

### 改名范围

当前默认只允许对上传到知识库的文件改名。

如果目标文件属于固定知识库材料：

- 默认明确说明当前不直接改固定材料文件名
- 不应假装已经完成改名

## 知识库文件导出

### “把知识库里的某篇 PDF 发给我”

默认先确认意图，不直接发文件。

默认先澄清：

- 是要原 PDF 文件
- 还是要摘要 / 主要内容

如果用户明确说“我要原文件”：

- 默认直接把 PDF 发回聊天

如果用户明确说“我要摘要”：

- 默认先问是直接在聊天里给摘要，还是顺手生成文档

### “把知识库里最相关的那篇发给我”

默认流程：

1. 系统先匹配最相关文件
2. 再做确认

“最相关”的默认依据：

- 文件名
- 最近对话上下文
- 当前绑定文档 / 最近上传文件

如果匹配出多个较相关文件：

- 列出前几个候选
- 让用户选择并确认

如果用户明确说：

- “就第 2 个，发原文给我”

则视为“对象 + 动作”都已明确，默认直接执行，不再重复确认。

如果用户在候选后回复：

- “摘要和文档都要”

默认先生成文档，再回一句摘要。

## “把这份文档加入知识库”

如果指代不清，系统默认不猜测。

应优先引用最近一个候选对象并确认：

- 最近上传的文件
- 最近消息里出现的文档链接
- 当前会话里最近绑定的文档

确认格式：

“请问您说的是这份文档吗？”

并附上文件名、标题或链接标识。

如果存在多个候选对象，不按固定类型优先，而按“最近一次被用户显式引用或上传的对象”作为首候选；若仍不清楚，再列出多个候选让用户选择。

## “把知识库里的文档加入当前文档”

### 目标文档与来源文档

如果用户说：

- “把知识库里那篇文档加入当前正在编辑的文档”

默认先确认目标文档，再确认来源文档。

即使目标文档已经比较明确，也默认把目标文档顺手带一句确认出来，再确认来源文档。

如果来源文档和目标文档都不明确：

- 默认仍按“先目标、后来源”的顺序确认

### “加入”的动作方式

如果用户只说“加入”，默认先澄清“加入”的方式，不擅自理解成贴原文或只写摘要。

如果用户明确说：

- “把知识库里的第 2 篇总结后加入当前文档”

默认仍再确认一次。

这次确认默认一次性确认三件事：

- 目标文档
- 来源文档
- 动作方式

### 加入位置

如果用户没有说明加到哪里：

- 默认先问要加到哪里

如果用户明确说：

- “加到文档最后”

则默认直接执行。

如果用户说：

- “你看着加吧”

则默认按当前文档结构自动选择位置。

自动选择位置时，默认优先：

- 加到最相关章节后面

## “把知识库里的文档替换掉当前文档里的相关部分”

如果用户说：

- “把知识库里的某篇文档替换掉当前文档里的相关部分”

默认先澄清“替换哪一部分”。

如果用户说：

- “你自己看着替换吧”

则默认按当前文档结构自动找最相关部分。

自动找相关部分时，默认优先匹配：

- 章节标题
- 相邻正文

如果系统已经找到将被替换的部分：

- 默认先展示将被替换的部分
- 再做确认

如果用户回复：

- “行，就这样”

则默认直接执行替换。

## “把知识库里的第 N 篇扩写成当前文档的一节”

如果用户说：

- “把知识库里的第 2 篇扩写成当前文档的一节”

默认规则：

- 当前文档结构明显时，自动选最相关章节
- 当前文档结构不明显时，先让用户指定章节

如果用户说：

- “那你新建一节吧”

默认先确认新章节标题。

如果用户说：

- “你自己起个标题吧”

则默认由系统自动生成标题。

自动生成标题时：

- 默认根据当前文档风格和知识库内容决定标题风格
- 如果当前文档标题风格很明显，默认优先跟随现有标题风格
- 默认先展示标题再确认

如果用户回复：

- “行，就这个”

则默认直接执行。

## 知识库文件删除

### 删除知识库文件

默认流程不是直接删除，而是：

1. 先按文件名列出匹配项
2. 让用户选择具体文件
3. 再做一次确认
4. 确认后才删除

### “删掉最相关那篇”

如果用户说：

- “把知识库里和 XXX 最相关的文档删掉”

默认流程是：

1. 先匹配最相关文件
2. 再确认删除

即使只匹配到一个文件，也要先展示匹配文件，再确认删除。

## 文档指代与确认

### “帮我改一下刚才那个文档”

默认先引用当前最可能的绑定文档做确认，而不是直接修改。

确认格式：

“请问您说的是这份文档吗？”

并附上 `doc_name` 或 `doc_url` 标识。

当前阶段默认先确认。

### “重新生成一份文档”

默认先确认一次是否要新建文档。

如果用户连续强调：

- 重新生成
- 新建
- 不要改旧文档

则不再反复确认，直接按新建处理。

## 两段式回复

当流程较长且意图明确时，默认采用两段式回复：

1. 先给一条短确认
   - 例如：已收到，正在处理……
2. 完成后再给最终结果

## 智能表格

### 基于知识库生成智能表格

如果用户明确要求：

- 生成智能表格
- 基于知识库生成智能表格
- 把知识库内容整理成智能表格

则主意图应识别为：

- `intent_family = smartsheet`
- `intent = smartsheet.create`

其中“知识库”只是数据来源范围，不应把整条请求路由成 `kb.list`。

### 动作边界

智能表格意图进入 `smartsheet` 家族后：

- 允许企业微信文档 / 智能表格相关工具
- 不允许自由掉入 `rag.*`
- 不应先退化成单纯列知识库文件
