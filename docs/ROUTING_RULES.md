# 路由规则

## 目的

这份文档定义：

- 输入先按什么维度分流
- 什么情况先澄清
- 什么情况直接执行
- 什么情况必须确认后再执行

## 两层分流

### 第一层：消息载体类型

优先区分：

1. 文本消息
2. 链接
3. PDF
4. 图片
5. 其他

说明：

- `链接` 主要指文本消息里包含的链接，尤其是企微文档链接
- `PDF` 指文件上传消息
- `图片` 当前阶段不重点支持，但仍应先识别为图片消息，而不是误当成文本或知识库材料

### 第二层：文本语义类型

只在“文本消息”里继续细分：

1. 普通文本
2. 包含企微文档链接的文本
3. 包含其他链接的文本
4. 纯指令文本

## 默认主流程

- 文本消息
  - 先让 agent 理解
- 企微文档链接
  - 先判断是不是企微文档链接，再判断用户意图
- PDF
  - 先入知识库
- 图片
  - 当前阶段不重点处理，必要时先澄清
- 其他
  - 先澄清

## 企微文档链接

### 链接 + 意图不明确

如果用户给了企微文档链接，但意图不明确，默认先澄清，不直接编辑、总结或重写。

例如：

- “看一下这个”
- “这个怎么弄”
- “帮我处理一下”

### 澄清方式

默认采用“半结构化 + 开放补充”：

- 先给 2 到 4 个高频意图选项
- 再给一个开放补充口子
- 不要求用户只能回复编号

推荐选项：

1. 总结这份文档
2. 修改这份文档
3. 读取这份文档的主要内容

“读取主要内容”指给结构概览或简要摘要，不是只发链接，也不是默认贴全文。

### 链接与上下文

即使消息里有企微文档链接，系统也只能把它当候选对象，不能代替明确意图。

系统在澄清前，应先检查：

- 当前会话里是否已有绑定文档
- 当前链接是否就是当前绑定文档
- 最近一次用户意图是什么

如果意图仍不明确，默认先澄清。

## PDF 上传

### 用户只上传 PDF，不说话

默认流程：

1. 先确认文件已加入知识库
2. 再主动询问下一步意图
   - 是否需要总结
   - 是否需要加入现有文档
   - 是否需要生成新文档

说明应使用用户能懂的话，不讲技术实现细节。

### 上传后只说“帮我总结一下”

默认不擅自决定输出形式，先澄清：

- 直接在聊天里总结
- 生成一份企微文档
- 或者两者都要

### 上传后明确说“总结并生成文档”

如果用户明确说：

- “帮我总结一下，并生成文档”
- “把刚上传的 PDF 总结一下，并生成文档”

则视为意图明确：

1. 不再额外澄清
2. 先确认文件已可用
3. 直接执行总结 + 生成文档

## 知识库文件管理

### 询问知识库里有几篇文章

默认返回：

- 总数量
- 按顺序列出前 10 篇文件名

如果用户明显只想要简略答案，可以只给总数，并说明如有需要可以继续列出前 10 篇。

不默认附带简介；只有用户继续追问时，再补每篇简介或摘要。

### “把这份文档加入知识库”

如果指代不清，系统默认不猜测。

应优先引用最近一个候选对象并确认：

- 最近上传的文件
- 最近消息里出现的文档链接
- 当前会话里最近绑定的文档

确认格式：

“请问您说的是这份文档吗？”

并附上文件名、标题或链接标识。

如果存在多个候选对象，不按固定类型优先，而按“最近一次被用户显式引用或上传的对象”作为首候选；若仍不清楚，再列出多个候选让用户选择。

### 删除知识库文件

默认流程不是直接删除。

而是：

1. 先按文件名列出匹配项
2. 让用户选择具体文件
3. 再做一次确认
4. 确认后才删除

## 文档指代与确认

### “刚才那个文档”“上一个文档”

默认先引用当前最可能的绑定文档做确认，而不是直接修改。

确认格式：

“请问您说的是这份文档吗？”

并附上 `doc_name` 或 `doc_url` 标识。

当前阶段默认先确认。后续如果用户已经连续多轮都在操作同一份文档，再考虑放宽。

### “重新生成一份文档”

默认先确认一次是否要新建文档。

如果用户已经连续强调：

- 重新生成
- 新建
- 不要改旧文档

则不再反复确认，直接按新建处理。
