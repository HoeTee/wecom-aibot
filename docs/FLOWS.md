# 流程

## 目的

这份文档定义关键流程的默认顺序。

重点不是覆盖所有情况，而是先固定当前最常见、最重要的流程。

## 流程 1：用户只上传 PDF

1. 接收 PDF 文件消息
2. 校验文件类型
3. 将 PDF 加入知识库
4. 向用户确认文件已加入知识库
5. 继续询问下一步意图

当前阶段不应：

- 把 PDF 先当普通文本处理
- 在未确认入库前就声称后续总结已经完成

## 流程 2：上传 PDF 后，用户要求“加入知识库”

1. 找到最近上传文件状态
2. 确认该文件已加入知识库
3. 直接回复确认结果

当前阶段不应：

- 再次索要文件
- 再次要求用户上传
- 把这条确认消息重新丢给 agent 自由问答

## 流程 3：上传 PDF 后，用户要求“总结并生成文档”

1. 确认文件已可用
2. 向用户回一条短确认，说明开始处理
3. 检索或总结文件内容
4. 生成结构化文档内容
5. 创建新文档
6. 返回最终结果

当前阶段不应：

- 意图已经明确还继续澄清
- 还没完成文档生成就声称已经完成

## 流程 4：文本消息中包含企业微信文档链接，但意图不明确

1. 识别出文档链接
2. 检查当前会话最近上下文
3. 如果意图仍不明确，进入澄清分支
4. 给出半结构化选项
5. 等待用户确认真实意图

当前阶段不应：

- 直接编辑文档
- 直接总结文档
- 直接覆写文档

## 流程 5：用户说“看一下这个”

如果消息中包含文档链接，但只有“看一下这个”这类模糊指令：

1. 视为意图不明确
2. 默认先澄清
3. 给出高频选项：
   - 总结
   - 修改
   - 读取主要内容

当前阶段不应擅自假设是“总结”还是“修改”。

## 流程 6：用户说“帮我改一下刚才那个文档”

1. 读取当前会话里的最近绑定文档
2. 先引用这份文档做确认
3. 用户确认后，再进入修改流程

当前阶段默认先确认。

## 流程 7：用户说“重新生成一份文档”

1. 判断这是 fresh document request
2. 先确认是否真的要新建文档
3. 如果用户继续明确强调“新建 / 重新生成 / 不要改旧文档”
4. 直接按新建流程处理

当前阶段不应：

- 未确认就默认复用旧文档
- 未确认就静默新建新文档

## 流程 8：知识库列表查询

1. 识别用户是在问知识库文件列表
2. 直接走知识库枚举逻辑
3. 返回总数
4. 默认列出前 10 篇文件名

当前阶段不应：

- 误路由到 RAG 总结
- 误路由到文档编辑

## 流程 9：知识库相关性查询

### 用户问“有没有和 XX 相关的文档”

1. 先按文件名、元信息做匹配
2. 如果没有候选，或候选太多、置信度不够，再进入 RAG 检索
3. 如果进入了 RAG，先给一句总判断
4. 再列出候选文档
5. 让用户决定下一步操作

当前阶段不应：

- 一上来就直接走 RAG
- 在相关性不明确时直接替用户决定具体文档

### 用户问“最适合回答这个问题的文档”

1. 先做文件名 / 上下文 / 当前绑定对象的综合匹配
2. 返回前 3 个候选
3. 标出最推荐的一个
4. 给一句简短推荐理由

如果用户后续只说：

- “就看第 2 个”

默认进入下一步动作澄清：

- 是要摘要
- 还是要原文
- 还是要生成文档

如果用户后续只说：

- “这个”

默认只引用当前候选列表里的第一个候选做确认。

## 流程 10：导出知识库文件

### 用户说“把某篇 PDF 发给我”

1. 先澄清用户要的是原文件还是摘要/主要内容
2. 如果用户明确要原文件，直接回传 PDF
3. 如果用户明确要摘要，先问是聊天内摘要还是顺手生成文档

### 用户说“把最相关的那篇发给我”

1. 先按文件名、最近上下文、当前绑定文档/最近上传文件匹配最相关文件
2. 如果只有一个高置信候选，先确认
3. 如果有多个相关候选，列出前几个让用户选择并确认

如果用户明确说：

- “就第 2 个，发原文给我”

默认直接执行，不再重复确认。

如果用户说：

- “摘要和文档都要”

默认先生成文档，再回一句摘要。

当前阶段不应：

- 还没确认就直接发送错误文件
- 还没澄清就默认把摘要当原文件返回

## 流程 11：删除知识库文件

1. 识别用户意图是删除
2. 先按文件名或相关性列出候选项
3. 让用户选择具体文件
4. 再做一次确认
5. 确认后才执行删除

如果用户说“删掉最相关那篇”：

1. 先匹配最相关文件
2. 展示匹配结果
3. 再确认删除

当前阶段不应：

- 未匹配候选就删除
- 未确认就删除

## 流程 12：知识库文件中的上传文件识别

1. 当用户问“哪个是我刚上传的文档”时，优先读取最近上传记录
2. 如果最近有明确上传记录，默认只回答最近一个上传文件
3. 如果最近没有明确上传记录，列出可能候选文件让用户确认

当前阶段不应：

- 在没有上传记录时凭空指定某一个文件
- 默认给上传文件额外打标而不经过用户询问

## 流程 13：把知识库文档加入当前文档

如果用户说：

- “把知识库里的某篇文档加入当前正在编辑的文档”

默认流程：

1. 先确认目标文档
2. 再确认来源文档
3. 再确认“加入”的动作方式
4. 用户确认后才执行

如果用户明确说：

- “把知识库里的第 2 篇总结后加入当前文档”

默认仍做一次整体确认。

如果目标文档已经比较明确：

- 也默认顺手把目标文档带出来确认

如果用户没有说明加到哪里：

- 默认先问要加到哪里

如果用户明确说：

- “加到文档最后”

默认直接执行。

如果用户说：

- “你看着加吧”

默认按当前文档结构自动选择位置，并优先加到最相关章节后面。

当前阶段不应：

- 在来源和目标都不清楚时直接执行
- 擅自把“加入”理解成贴全文或只写摘要

## 回复节奏

当流程较长且意图明确时，默认采用两段式回复：

1. 先发一条短确认
   - 例如：已收到，正在处理……
2. 完成后再发最终结果
