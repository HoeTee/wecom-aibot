# 流程

## 目的

这份文档定义关键流程的默认顺序。

重点不是覆盖所有情况，而是先固定当前最常见、最重要的流程。

## 流程 1：用户只上传 PDF

1. 接收 PDF 文件消息
2. 校验文件类型
3. 将 PDF 加入知识库
4. 向用户确认文件已加入知识库
5. 继续询问下一步意图

当前阶段不应：

- 把 PDF 先当普通文本处理
- 在未确认入库前就声称后续总结已经完成

## 流程 2：上传 PDF 后，用户要求“加入知识库”

1. 找到最近上传文件状态
2. 确认该文件已加入知识库
3. 直接回复确认结果

当前阶段不应：

- 再次索要文件
- 再次要求用户上传
- 把这条确认消息重新丢给 agent 做自由问答

## 流程 3：上传 PDF 后，用户要求“总结并生成文档”

1. 确认文件已可用
2. 向用户回一条短确认，说明开始处理
3. 检索或总结文件内容
4. 生成结构化文档内容
5. 创建新文档
6. 返回最终结果

当前阶段不应：

- 意图已经明确还继续澄清
- 还没完成文档生成就声称已经完成

## 流程 4：文本消息中包含企微文档链接，但意图不明确

1. 识别出文档链接
2. 检查当前会话最近上下文
3. 如果意图仍不明确，进入澄清分支
4. 给出半结构化选项
5. 等待用户确认真实意图

当前阶段不应：

- 直接编辑文档
- 直接总结文档
- 直接覆写文档

## 流程 5：用户说“看一下这个”

如果消息中包含文档链接，但只有“看一下这个”这类模糊指令：

1. 把它视为意图不明确
2. 默认先澄清
3. 给出高频选项：
   - 总结
   - 修改
   - 读取主要内容

当前阶段不应擅自假设是“总结”还是“修改”。

## 流程 6：用户说“帮我改一下刚才那个文档”

1. 读取当前会话里的最近绑定文档
2. 先引用这份文档做确认
3. 用户确认后，再进入修改流程

当前阶段默认先确认。

## 流程 7：用户说“重新生成一份文档”

1. 判断这是 fresh document request
2. 先确认是否真的要新建文档
3. 如果用户继续明确强调“新建/重新生成/不要改旧文档”
4. 直接按新建流程处理

当前阶段不应：

- 未确认就默认复用旧文档
- 未确认就静默新建新文档

## 流程 8：知识库列表查询

1. 识别用户是在问知识库文件列表
2. 直接走知识库枚举逻辑
3. 返回总数
4. 默认列出前 10 篇文件名

当前阶段不应：

- 把它误路由到 RAG 总结
- 把它误路由到文档编辑

## 流程 9：删除知识库文件

1. 识别用户意图是删除
2. 先按文件名列出候选项
3. 让用户选择具体文件
4. 再做一次确认
5. 确认后才执行删除

当前阶段不应：

- 未匹配候选就删除
- 未确认就删除

## 回复节奏

当流程较长且意图明确时，默认采用两段式回复：

1. 先发一条短确认
   - 例如：已收到，正在处理……
2. 完成后再发最终结果
